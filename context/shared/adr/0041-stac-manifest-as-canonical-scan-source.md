# ADR-0041: STAC Manifest as Canonical Scan Source

## Status
Adopted

## Context

Two metadata-scanning code paths existed in parallel and disagreed on what
counts as a "thing that needs metadata":

1. `MetadataFreshRule.check()` walked the filesystem with an extension
   filter (`.parquet`, `.tif`, `.tiff`) and required every match to have a
   sibling STAC item JSON. It emitted `MISSING` for any unmatched file.
2. `check_directory_metadata()` walked existing `*.json` files only. It
   could emit `FRESH`, `STALE`, or `BREAKING`, but **never** `MISSING`.

The `--fix` flow consumed only the second pipeline's report, so:

- `check` would report `12 missing` for a catalog.
- `check --metadata --fix` would say `5 items already fresh` and apply
  no fix ŌĆö the `MISSING` results from path 1 never reached path 2.
- Re-running `check` would report the same 12 missing files.

This produced the user-visible confusion documented in **issue #384**.

The same filesystem-walk path also broke for collection-level rollup
assets like `items.parquet` (generated by `add --stac-geoparquet`).
Per ADR-0031, collection-level vector assets and rollup indexes are
registered in `collection.json.assets` and have **no companion item JSON
by design**. The rule treated them as items-needing-JSON anyway, producing
permanent false `MISSING` reports ŌĆö **issue #345**.

Two divergent scanners with different type domains cannot be reconciled
by tweaking either one. The asymmetry is structural.

## Decision

The STAC manifest tree is the canonical source of truth for "what assets
exist and need metadata." A single scanner ŌĆö `scan_catalog_metadata` in
`portolan_cli/metadata/scan.py` ŌĆö walks `catalog.json` ŌåÆ child links ŌåÆ
`collection.json` ŌåÆ `item.json`, and at each registered asset performs the
appropriate freshness check. Both `MetadataFreshRule.check()` and the
`portolan check --fix` flow consume the `MetadataReport` produced by this
single scanner.

### What the scanner emits

For every registered asset (collection-level *or* item-level), one of:

- `FRESH` ŌĆö versions.json/heuristics agree with current file state.
- `STALE` ŌĆö file mtime/heuristics differ from stored values.
- `BREAKING` ŌĆö schema fingerprint changed (column removed, type altered).
- `MISSING` ŌĆö item directory contains data file(s) but no `<item_id>.json`,
  *or* an asset registered in item.json no longer exists on disk.

For files on disk that are **not** referenced by any manifest, the scanner
emits a fifth status:

- `ORPHANED` ŌĆö data file under a collection (or item dir) that is not
  registered in `collection.json.assets` or any `item.json.assets`. Not
  auto-fixable. Reported as a `WARNING` with a fix hint instructing the
  user to either register it via `portolan add` or delete it.

### Layout assumptions (matches `add` output)

- Item JSON: `{collection_dir}/{item_id}/{item_id}.json` (hierarchical).
- Item assets: `{collection_dir}/{item_id}/<asset>` (sibling to item.json).
- Collection-level vector single-file (ADR-0031): `{collection_dir}/data.parquet`
  registered in `collection.json.assets`. No item.json expected.
- Items.parquet rollup (`add --stac-geoparquet`): `{collection_dir}/items.parquet`
  registered as collection asset under key `geoparquet-items`.
- Pattern 2 sub-catalogs (ADR-0032): `{collection_dir}/{sub-catalog}/catalog.json`
  with item subdirs underneath. Items still belong to the enclosing
  collection; `versions.json` is resolved against the collection root.

### What `check_directory_metadata` is for now

`portolan_cli/metadata/validation.py:check_directory_metadata` is retained
exclusively for **STAC structural validation** (catalog/collection/item
JSON shape). It is no longer used as input to the fix flow.

## Consequences

**Easier:**

- `check` and `check --fix` are guaranteed to agree by construction ŌĆö
  same input, same `MetadataReport`. The asymmetry class of bugs is
  closed.
- New asset patterns (e.g., future rollup formats, sidecar registries)
  inherit correctness automatically as long as they declare themselves
  in `collection.json` or `item.json`.
- Orphan detection is now a first-class output, surfacing files that
  would otherwise remain invisible to validation.

**Harder:**

- Catalogs that violate the layout (e.g., flat item.json next to data
  file in collection root) are detected as ORPHANED rather than treated
  as legacy items. `get_stored_metadata` falls back to the flat layout
  for backward compatibility, but the orphan sweep still flags the data
  file unless the user converts to the hierarchical layout.
- The scanner relies on filename convention `{item_id}.json` matching
  the parent dir name. This matches what `add` produces today; future
  changes to `add`'s naming would need scanner updates.

## Alternatives considered

### Tactical allow-list in `MetadataFreshRule` only

Skip a hardcoded set of filenames (`items.parquet`) when walking the
filesystem. Rejected: closes #345 but not #384, since the two-pipeline
asymmetry remains. Also fragile ŌĆö every new collection-level asset would
require an allow-list update.

### Migrate fix flow to consume `MetadataFreshRule`'s rglob output

Rejected: the rglob output never sees the STAC structural validation
that `check_directory_metadata` performs, and the rglob walk has its own
correctness problems (the items.parquet false MISSING). Reusing a broken
scanner does not help.

### Walk only item directories (no manifest reading)

Rejected: per ADR-0031, vector single-file collections have NO item
directories at all. Walking item dirs alone misses collection-level
assets entirely.
